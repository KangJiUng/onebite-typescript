## 섹션 1: 타입스크립트 개론

- TypeScript의 창시자 === C#의 창시자

- 타입스크립트는 기존의 자바스크립트를 더 안전하게 사용할 수 있도록 타입 관련된 여러 가지 기능들을 추가한 언어. 쉽게 말해서 자바스크립트의 확장판
    - 자바스크립트에서는 변수의 타입을 프로그래머가 직접 정의할 수 있는 방법이 없는 반면에 타입스크립트 코드에서는 변수의 이름 뒤에 콜론과 함께 변수의 타입을 정의할 수 있다.
    - 자바스크립트 코드에서 타입을 정의하는 문법만 딱 추가하면 타입스크립트 코드

- TypeScript가 꼭 필요한가?
    
    → 자바스크립트는 아주 유연하게 설계, 버그로부터의 안정성이나 견고함은 일부 포기하는 대신 프로그래머가 쉽고 빠르고 간결하게 코드를 작성하는데에 중점을 두고 만들어짐
    
    → 자바스크립트의 런타임 Node.js 탄생 이후에는 웹 브라우저에서만 실행할 수 있었던 자바스크립트를 이제는 어디서든 실행 가능하게 되면서 프로그래머들이 자바스크립트를 이용해 다양한 프로그램을 만들기 시작
    
    → 복잡한 대규모 어플리케이션을 자바스크립트로 개발하다 보니 오히려 유연함이 프로그램의 안정성을 떨어뜨리는 단점이 되어 버린 것
    
    → 이런 자바스크립트의 문제점을 극복하기 위해서 자바스크립트의 기존 문법들과 매력은 그대로 유지한 채 <ins>**안정성만 추가로 확보한 새로운 언어가 필요**</ins>해짐
    
    → 타입스크립트의 탄생
    

- **모든 프로그래밍 언어에는 타입 시스템이 있다.**
    - 타입 시스템: 프로그래밍 언어를 사용할 때 타입과 관련해서 지켜야하는 규칙들을 모아둔 체계. 즉, 언어의 타입 관련된 문법 체계
        - 정적 타입 시스템: 코드 실행 전에 모든 변수의 타입을 고정적으로 결정
        - 동적 타입 시스템: 코드를 실행하기 전에는 타입을 결정하지 않고 코드를 실행하고 나서 그때그때 유동적으로 변수의 타입을 결정

- 동적 타입 시스템(feat. JavaScript)
    - 장점: 우리가 미리 변수에 타입을 설정하지 않아도 된다는 유연함
        - 변수의 타입이 어떤 하나의 타입으로 딱 고정되지 않고 현재 변수에 담긴 값에 따라서 변수의 타입이 동적으로 계속해서 달라짐
    - 단점: 타입 에러가 발생해도 코드 실행 자체는 되기 때문에 오류가 실행과 동시에 발생하지 않을 수 있음(→ 런타임 오류 발생 가능)

- 정적 타입 시스템(feat. Java)
    - 장점: 코드 실행 전에 타입 관련 에러가 있다면 미리 보여줌
        - 타입을 잘못 쓰지 않았는지 검사까지 모두 마치고 실행되기 때문에 오류가 있다면 애초에 실행이 불가능
    - 단점: 모든 변수에 일일히 타입을 다 정의해야하기 때문에 귀찮고 작성해야하는 코드의 양이 매우 늘어남

- **점진적 타입 시스템**(Gradual Type System)(feat. **TypeScript**)
    - 변수의 타입을 우리가 직접 정의 하지 않아도 변수에 담기는 초기값을 기준으로 자동으로 타입을 알아서 추론
    - 타입이 정의된 변수들에 대해서는 타입을 미리 결정하고 타입이 정의되지 않은 변수들에 대해서는 타입을 자동으로 추론하는 타입 시스템

- 대다수의 프로그래밍 언어는 어떻게 동작할까?
    - 대부분의 프로그래밍 언어는 컴퓨터보단 인간에게 더 친화적 → 컴퓨터는 코드를 자기가 해석하기 쉬운 형태로 변환(컴파일, compile)
        - 컴파일러: 코드를 컴파일하는 역할
    - 자바나 자바스크립트같은 언어를 컴파일하면 바이트 코드라는 형식으로 변환
        - 바로 바이트코드로 변환하는게 아니라 그 전에 AST(추상 문법 트리)라는 특별한 형태로 먼저 변환
            <img width="1108" height="300" alt="image" src="https://github.com/user-attachments/assets/b418e633-ed12-4552-b0d6-cdd4571a1512" />

            - AST(추상 문법 트리): 코드의 공백이나 주석 탭 등의 코드 실행에 관계없는 그런 요소들은 전부 제거하고 트리 형태의 자료구조에 코드를 쪼개서 저장 해 놓은 형태

- 타입스크립트의 동작 과정
    - 코드를 먼저 AST로 변환(다른 언어와 동일) → **타입 검사** 수행 → 바이트코드가 아니라 <ins>**‘자바스크립트 코드’로 변환**</ins>
        - 타입 오류가 발생하고 있는 타입스크립트 코드는 컴파일 시 타입 검사를 통과할 수 없기 때문에 자바스크립트 코드로 변환되지 않아 실행할 수 없게 되는 것
        - 타입 오류가 없는 타입스크립트 코드는 컴파일 시 타입 검사를 통과해서 <ins>**타입 관련 문법들은 삭제**</ins>된 안전한 자바스크립트 코드로 변경 → 타입 관련 코드들은 결국 바스크립트로 변환될 때 사라지게 되어 프로그램 실행에 영향을 미치지 않는다

- 컴파일러 옵션
    - 얼마나 엄격하게 타입 오류를 검사할 건지 또는 컴파일 결과 생성되는 자바스크립트 코드의 버전은 어떻게 할 것인지 등의 컴파일의 아주 세부적인 사항들
    - 타입스크립트는 다른 언어에 비해서 컴파일러 옵션을 아주 자유롭고 쉽게 설정이 가능

- 컴파일러 옵션의 종류(tsconfig.json)
    - include 옵션
        - tsc(타입스크립트 컴파일러)에게 컴파일 할 타입스크립트 파일의 범위와 위치를 알려주는 옵션
    - target 옵션
        - 컴파일 결과 생성되는 자바스크립트 코드의 버전을 설정하는 옵션
        - 예전 JavaScript 버전을 사용해야할 때가 은근 있음. 중요한 옵션
        - include 옵션과 달리 compilerOptions 안에 들어있는 이유는 자바스크립트로 변환하는 과정이나 타입 검사 등에 아주 상세한 옵션들을 설정하기 위한 것
    - module 옵션
        - 변환되는 자바스크립트 코드의 모듈 시스템을 설정하는 옵션
    - outDir 옵션
        - 컴파일 결과 생성할 자바스크립트 코드의 위치를 결정하는 옵션
    - strict 옵션
        - 입스크립트 컴파일러의 타입 검사 엄격함 수준을 정하는 옵션
        - 타입스크립트는 함수 매개변수의 타입을 자동 추론할 수 없기 때문에 타입을 프로그래머가 직접 지정하지 않을 경우 엄격한 타입 검사 모드에서는 오류가 발생
    - ModuleDetection 옵션
        - 옵션을 force로 설정할 경우 자동으로 모든 타입스크립트 파일이 로컬 모듈(독립 모듈)로 취급
        - 타입스크립트의 모든 파일은 기본적으로 전역 파일(모듈)로 취급되기 때문에 다음과 같이 a.ts와 b.ts 두 타입스크립트 파일을 만들고 동일한 이름의 변수를 선언하면 오류가 발생함
            - 이럴 때에는 <ins>**각 파일에 모듈 시스템 키워드(export, import)를 최소 하나 이상 사용해**</ins> 해당 파일을 전역 모듈이 아닌 로컬(독립) 모듈로 취급되도록 만들어야 하는데 이를 자동화 하는 옵션이 바로 moduleDetection 옵션
        ```tsx
        // a.ts
        const a = 1; // ❌
        
        // b.ts
        const a = 1; // ❌
        ```
